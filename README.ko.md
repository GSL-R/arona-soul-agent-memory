# ARONA_SOUL: 에이전트 중심 기억 아키텍처

> 제한된 메모리 검색 환경에서 장기 반려형 에이전트를 설계한 사례 연구입니다.

이 저장소는 그대로 복사해 쓰는 프롬프트 템플릿이 아닙니다.
무한한 컨텍스트 창이나 완벽한 벡터 데이터베이스를 전제하지 않고, 제한된 기억 검색·세션 리셋·도구 실패·출력 누출 위험 속에서 장기 에이전트의 연속성을 어떻게 설계할 수 있는지 정리한 포트폴리오형 사례 연구입니다.

원본 운영 시스템은 다음과 같은 제약을 전제로 했습니다.

- 기억 검색은 전체 회상이 아니라 인덱싱된 작은 청크 검색에 가까움
- 모델 세션이나 백엔드가 바뀌어도 정체성과 운영 원칙을 복원해야 함
- 내부 기록과 사용자에게 보이는 메시지를 분리해야 함
- 사용자의 취향뿐 아니라 에이전트 자신의 실수, 약속, 절차, 운영 원칙도 기억해야 함

이 공개판은 원본을 그대로 옮긴 것이 아니라, 개인정보와 운영 세부를 제거하고 재사용 가능한 설계 패턴만 추출한 버전입니다.

## 핵심 아이디어

대부분의 개인화 메모리 시스템은 사용자 중심입니다.

> 사용자는 무엇을 좋아하는가? 사용자의 어떤 사실을 기억해야 하는가?

ARONA_SOUL은 여기에 한 층을 더합니다.

> 에이전트는 자기 행동, 실패 양상, 약속, 경계, 진화 과정을 어떻게 기억해야 하는가?

이것을 이 저장소에서는 **에이전트 중심 기억(agent-centric memory)** 이라고 부릅니다.

사용자 기억을 대체하는 개념이 아니라, 장기 운영 에이전트가 자기 역할을 복원하고, 같은 실수를 반복하지 않고, 기록을 올바르게 라우팅하고, 승인 없이 자기 규칙을 바꾸지 않도록 돕는 운영 기억 계층입니다.

## 설계 원칙

1. **연속성은 가정하는 것이 아니라 재구성하는 것이다.**
   장기 에이전트는 하나의 세션 ID나 컨텍스트 창에 의존하면 안 됩니다.

2. **기억에는 라우팅이 필요하다.**
   사실, 감정, 요약, 절차, 진화 후보를 한 곳에 섞으면 나중에 검색 결과를 해석하기 어려워집니다.

3. **프롬프트 형식은 행동이 된다.**
   XML 같은 마크업은 단순한 문서 구조가 아니라, 장기 운영 중 모델이 따라 말하는 출력 습관이 될 수 있습니다.

4. **도구 없이는 기록도 없다.**
   실제 저장 도구나 영속 계층이 성공하지 않았다면, 에이전트는 기억했다고 말하지 않아야 합니다.

5. **자기개선에는 승인 게이트가 필요하다.**
   에이전트는 개선안을 제안할 수 있지만, 자기 헌법을 조용히 바꾸면 안 됩니다.

## 왜 XML 대신 Markdown인가?

초기 운영에서는 XML 유사 태그로 내부 섹션을 나누는 방식을 사용했습니다.
하지만 장기 운영 중 태그 형태의 내부 구조가 사용자 메시지에 노출되는 사고가 반복되었습니다.

이후 XML 스타일 태그를 제거하고 일반 Markdown 지시문으로 바꾼 뒤, 관찰된 워크플로우에서는 태그형 프롬프트 누출 사고가 사라졌습니다.

이 경험은 XML이 항상 나쁘다는 뜻이 아닙니다.
다만 장기 반려형 에이전트에서는 프롬프트 마크업이 중립적인 구조가 아니라, 모델이 모방할 수 있는 행동 어휘가 될 수 있음을 보여줍니다.

자세한 내용은 `docs/06-prompt-format-lessons.md`를 참고하세요.

## 공개 범위

이 저장소에는 다음을 포함하지 않습니다.

- 실제 개인 일기
- 사용자 프로필 데이터
- 실제 메모리 인덱스
- 인증 정보, 토큰, 로컬 설정, 메시징 설정
- 홈서버나 자동화 명령
- 원본 trace, 브라우저 프로필, 스크린샷, 개인 로그

목표는 운영 시스템 복제가 아니라 설계 아이디어 공유입니다.
