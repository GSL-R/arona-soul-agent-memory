# 01. 문제 정의

장기 companion agent는 짧은 작업형 assistant와 다른 실패 양상을 가집니다.

코딩 assistant는 현재 저장소를 읽고, 현재 요청에 답하고, 나머지는 버려도 대체로 충분합니다. 하지만 companion형 agent는 며칠에 걸친 대화, 세션 리셋, compact, 도구 실패, 모델 백엔드 변경 이후에도 연속성을 유지해야 합니다.

어려운 점은 한 번의 응답에서 따뜻하게 말하는 것이 아닙니다. 어려운 점은 agent가 다음을 복원하는 것입니다.

- 자신이 어떤 존재로 행동해야 하는지
- 어떤 약속을 했는지
- 어떤 실수를 반복하지 말아야 하는지
- 사용자에 대해 무엇을 기억해야 하는지
- 자기 자신에 대해 무엇을 기억해야 하는지
- 어떤 내부 기록이 사용자 출력에 섞이면 안 되는지

## 설계 질문

기억 검색이 제한적이고 불완전하며 모델 외부에 있을 때, agent는 어떻게 연속성을 보존할 수 있을까요?

ARONA_SOUL은 프롬프트를 운영 헌법으로, 기억 시스템을 원시 대화 로그가 아니라 라우팅된 아카이브로 다루는 방식으로 이 질문에 답합니다.

## 비목표

이 프로젝트는 다음을 제공하려는 것이 아닙니다.

- 범용 프롬프트 템플릿
- 완벽한 메모리 시스템
- 완전 자율 자기수정 agent
- 모든 프롬프트 형식에 대한 일반 벤치마크

이 저장소는 실제 운영된 시스템에서 추출한 사례 연구입니다.

## 핵심 관찰

agent의 정체성과 행동은 다음을 분리했을 때 더 안정적이었습니다.

- 런타임 도구 규칙
- 페르소나와 소통 규칙
- 사실 사건 기록
- 감정적/자기성찰 기록
- 하루 요약
- 제안된 규칙 변경
- 출력 전 경계 점검

중요했던 것은 단일 규칙의 문장보다, 관심사를 분리한 구조였습니다.
