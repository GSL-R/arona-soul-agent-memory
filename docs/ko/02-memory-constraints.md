# 02. 기억 제약

원본 시스템은 완벽한 회상을 전제하지 않았습니다.

이 시스템은 파일이 비교적 작은 청크로 인덱싱되고 검색을 통해 일부만 회수되는 제약 환경을 기준으로 설계되었습니다. 이런 환경에서 agent는 단순히 "모든 것을 기억한다"고 가정할 수 없습니다. 어디를 찾아야 하는지, 어떤 종류의 기억을 신뢰해야 하는지 알아야 합니다.

## 제약 모델

이 설계는 다음을 전제합니다.

- 기억은 모델 컨텍스트 외부에 저장됨
- 검색은 전체 생애사가 아니라 부분 청크를 반환함
- 비슷한 일상 이벤트가 서로 섞일 수 있음
- 오래된 기록은 낡은 해석을 포함할 수 있음
- 도구 실행은 실패할 수 있음
- 모델 세션은 리셋되거나 교체될 수 있음

이 제약들이 프롬프트 구조를 결정합니다.

## 설계 대응

프롬프트에는 명시적인 경로가 포함됩니다.

- 사건을 복원할 때는 사실 기록을 먼저 사용함
- 감정 일기는 정서적 맥락에만 사용함
- summary는 source of truth가 아니라 지도처럼 사용함
- 운영 규칙은 procedure 문서에서 확인함
- 장기 사용자 프로필로 승격하기 전에는 observation 단계를 둠

이는 검색 결과의 모호성에 대한 실용적 대응입니다.

## 왜 더 많이 저장하는 것만으로 부족한가?

기록이 타입 없이 쌓이면 메모리가 많을수록 검색 품질이 나빠질 수 있습니다.

사실 로그, 감정, 할 일, 사용자 선호, 자동화 결과, 자기성찰이 하나의 구분 없는 기억 공간에 섞이면 검색 결과를 해석하기 어려워집니다. agent가 시적인 성찰 기록을 검색해 사실 사건 로그처럼 취급할 수 있습니다.

기억 라우팅은 이를 방지하기 위해 존재합니다.

## 요점

제한된 메모리 시스템에서 프롬프트는 단순히 "이것을 기억하라"고 말하는 데 그치면 안 됩니다.
기억의 종류, 신뢰 수준, 검색 순서, 승격 규칙을 함께 정의해야 합니다.
